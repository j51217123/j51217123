CleanCode & 有意義的命名

發表於 2022-11-30 分類於 personal

前言

這個 Blog 擺放將近一年沒更新…不小心懶惰太久了哈哈，想到前陣子在公司分享過命名規範的主題，就整理一下放到自己的 Blog 上，廢話不多說，下面正文開始 !

什麼是 Clean Code ?

書中歸納了幾位程式大大對於 Clean Code 的看法，不外乎是淺顯易懂的、沒有贅餘重複、能通過單元及驗收測試，有著正規的錯誤處理機制。因為修改、撰寫程式前的第一步就是讀懂程式本身，花在閱讀程式的時間遠遠超過寫程式的時間。所以，所以讓程式碼更容易閱讀，也才能讓新的程式碼變得更容易撰寫。 我們在進行專案開發時，就可以順手從小地方開始做起，比如說清除一些重複或未使用的程式碼，又或是改變一個變數的名稱。

為什麼需要整潔、無瑕的程式碼 ?

這是一本關於「程式品質」的書，在不同的領域、不同的語言，定義「品質」的指標、原則可能不同。但雜亂而品質低劣的程式碼，所造成的影響效應是相近的，最中肯的評述，就是序言所提的『唯一有效的的『程式品質』度量單位： (Code Review時)每分鐘罵髒話的次數(wtfs/minutes)』 (個人覺得蠻中肯的XD

當我們持續寫出雜亂的程式碼時，會使團隊的產能持續下降，最終形成一個惡性循環，如果想要遏止這樣的情形發生，建議的方法是隨時隨地盡可能地確保程式碼整齊潔淨

何謂有意義的命名

就是讓名稱代表其意圖，不管是變數、函式甚至到元件，都要可以告訴我們，它是做什麼用，該如何使用，當我們命名具備展現意圖，就能更容易去理解及修改程式碼。

產生有意義的區別

  • 數字序列命名法

    • 雖然數字序列法可以正常被執行，但它帶來多少困擾，誰可以告訴我下圖中的 data1, data2 分別是什麼嗎 XD

    • 無意義的單字詞

      • 假如同時存在 ProductInfo 與 ProductData，其實是很難去區分兩者間的差別，就會變成是無意義的單詞

使用可被搜尋的名字

使用單一字母的名稱或數值，會造成搜尋上的困難，但也不是說長命名就是絕對，因為命名時需注意其長度應該與 Scope 相對應，試想下方中的數字 5 或 34，要如何在專案中精準搜尋，如果給予相對應的名稱，搜尋上就會輕鬆許多!

變數應使用名詞或是名詞片語來命名，例如 customer、wikiPage，同時避免使用 Info、Data 此類無區別意義的字詞

函式應使用動詞或動詞片語來命名，例如 postPayment、deletePage，取值或 call API 可以用 get、fetch、retrieve 當字首，修改時我們可以用 set 當字首，存值時可以用 onSave 當字首，判斷時可以用 is 當字首，且盡量在每次撰寫時使用一種持續到底，下方列出一些我從參考文章中我覺得還不錯的命名

總結

要挑選一個好的名稱，需要有良好的描述技巧，是需要長時間訓練的。開發過程中，不論是翻新命名或是新命名，透過細節上的處理，可以使程式碼的易讀性提高許多，或短或長的成果來看，都是正向的。
